CI/CD Pipeline — Github Actions

Context & Problem

There are two real problems to solve:

  • Avoid unnecessary builds when a change doesn't touch a service
  • Ensuring every service that does change goes through the same quality checks before reaching production

GitOps boundary -> CI owns building and tagging images; FluxCD owns deploying them.

Proposed Solution — Pipeline Structure

Shared reusable workflows

The Github Actions setup uses one reusable workflow in a single file, and triggers from each service's workflow file.

First, we require the service name and type as inputs. 
# .github/workflows/service-ci.yml  (reusable, called by each service)

on:
  workflow_call:
    inputs:
      service_name:
        required: true
        type: string
      service_type:
        required: true
        type: string  # 'api' | 'worker' | 'frontend'
The workflow has two jobs:
  • an affected-check job that computes whether the service is affected by the change
  • a ci job that runs the actual CI steps but only if the service is affected
jobs:
  affected-check:
    outputs:
      affected: ${{ steps.check.outputs.affected }}
    steps:
      - name: Checkout with full history
        ...
      - name: Compute Nx affected base/head SHAs
        ...
      - name: Check if service is affected
        ...
  ci:
    needs: affected-check
    if: needs.affected-check.outputs.affected == 'true'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install dependencies
        ...
      - name: Lint
        ...
      - name: Test
        ...
      - name: Build
        ...
      - name: Authenticate to Artifact Registry
        ...
      - name: Build and push container image
        ...
        # No :latest tag — SHA is the only reference used downstream

      - name: Promote image tag to dev
        if: github.ref == 'refs/heads/main'
        # Clones the GitOps repo
        # Updates envs/dev/apps//values.yaml image.tag to github.sha
        # Opens a commit — FluxCD reconciles from there
        # Int and above require a separate GitOps PR with human approval
Decision
I have not worked much with Nx, however from a quick crash course, I was able to identify that Nx's affected command is the key mechanism for skipping unnecessary builds. It computes the dependency graph from project.json files and only runs tasks for projects touched by the diff. This means a change to the observability stack doesn't trigger a rebuild of the storefront. You'll also notice that I've not used the input service_type in the workflow yet. This is intentional for the simplicity of my initial solution, but can be used later on for different build behaviour, test commands, health checks and deploy behaviour.

Image Tagging & Promotion

Image promotion flow. The SHA tag is immutable. Promotion is a GitOps PR that bumps the tag in the target environment's values.yaml.
  • Images are tagged with the commit SHA only. No :latest, no semver at this layer.

  • Every image is built once and promoted, never rebuilt per environment.

    Promotion works by opening a pull request in the GitOps repo that bumps the image tag in the target environment's values.yaml.

    • For dev, CI does this automatically on merge.
    • For int and above, it requires a PR and at least one human approval.
    • For prod, the approval gate is enforced by Github's required reviewers on the environment protection rules.

    • Preventing Broken Services from Reaching Prod

    The quality gate is a layered system, not a single check:

    • PR gate: lint, unit tests, and build must pass on every PR before merge. Branch protection rules enforce this.
    • Integration gate: after automatic deployment to int, a smoke test job runs against the live service. If it fails, the FluxCD HelmRelease is left in a Failed state and alerts fire. The tag is not promoted to QA.
    • QA gate: manual sign-off by QA or the service owner. This is a required reviewer on the PR, not a chat message.
    • Prod gate: required approval from a second engineer (not the author). Environment protection rules in Github enforce this even if the author has merge rights.
    • Post-deploy: FluxCD health checks (via HealthChecks on the HelmRelease) verify that the new pods reach Running state. If they don't within a timeout, Flux marks the release failed and Prometheus fires an alert. Rollback is a git revert of the tag bump PR.

    Kubernetes Platform Setup

    Resource Requests, Limits, and Autoscaling

    Resource requests are mandatory. Limits are more nuanced. CPU limits cause throttling even when the node has spare capacity, which is counterintuitive and leads to latency spikes. The approach here is different for CPU and memory (example values):

    Service type CPU request CPU limit Memory request Memory limit HPA
    NestJS API (10–3000 RPS) 200m none 512Mi 1Gi CPU 60%, min 2, max 20
    Next.js frontend (low load) 100m 500m 256Mi 512Mi CPU 70%, min 1, max 4
    NestJS worker (spikey) 250m none 1Gi 2Gi KEDA — Pub/Sub backlog
    • CPU limits are deliberately omitted for the API and worker services. CPU throttling at the container level creates latency spikes that look like application slowness.
    • Memory limits are in place to prevent a runaway container from consuming all node memory and causing OOM kills of other pods. The limits are set high enough that normal operation shouldn't hit them.
    • KEDA for workers. Workers are driven by Pub/Sub queue depth

    Abstractions for Service Deployment

    Every service is deployed via a shared Helm chart (charts/marketplace-service) that wraps Deployment, Service, HPA, ServiceAccount, PodDisruptionBudget, and NetworkPolicy into a single values interface. Teams interact only with values.yaml.

    # charts/marketplace-service/values.yaml  (what a service team sets)
image:
  repository: europe-west1-docker.pkg.dev/mms-prod/marketplace/order-service
  tag: abc1234

service:
  port: 3000
  type: api   # 'api' | 'worker' | 'frontend'

replicas:
  min: 2
  max: 15

env:
  NODE_ENV: production

envFromSecrets:
  - secretName: order-service-secrets
    keys: [DB_PASSWORD, JWT_SECRET]

resources:
  preset: api   # maps to the resource defaults in the table above

    The chart handles everything else: anti-affinity rules, topology spread constraints for zone distribution, the default NetworkPolicy allowing only intra-namespace + ingress-controller traffic, the ServiceAccount linked to Workload Identity, and the PodDisruptionBudget ensuring at least one replica stays up during node drain.

    Decision: Helm chart over CRD-based platform
    I like to keep tooling as lean as possible. There are probably other tools that could make life slightly easier, however, for 15 services and a small platform team, a well designed Helm chart with sensible defaults gets 95% of the value with 20% of the complexity.

    Guardrails — What to Add and What to Skip

    GuardrailAdd it?Rationale
    Admission webhook for resource limits no Requiring CPU limits creates throttling bugs. A webhook that blocks deployment without a CPU limit will cause incidents.
    ResourceQuota per namespace yes Prevents one misbehaving service from consuming all cluster resources. Set generously enough not to block normal operations.
    Pod Security Standards/Admission (restricted) warn mode Start in warn mode, fix violations, then enforce. Don't enforce before auditing as it will break services silently at deploy time.
    Network policies (default deny) yes Default deny-all with explicit allow for intra-namespace and ingress controller. Workers that need to reach Redis or Pub/Sub get explicit egress rules.
    OPA/Gatekeeper yes
    • Only images from the internal Artifact Registry are allowed.
    • No type: Loadbalancer

    Deployment Strategies

    Context & Problem

    The three service categories have fundamentally different risk profiles. A broken worker that processes payments differently than expected is a different kind of problem from a broken frontend component. Deploying all 15 services the same way is risky but this could also add unnecessary complexity.

    Strategy by Service Category

    Deployment strategy by service category. Each choice maps to the service's risk profile, not a preference for a particular pattern.

    NestJS APIs — Rolling update

    • Rolling update with maxUnavailable: 0 and maxSurge: 1 means new pods must be healthy before old ones are terminated. minReadySeconds: 30 adds a soak period before a pod is declared ready. This handles connection drain gracefully.
    • For releases that carry risk (database schema changes, major dependency bumps), a manual canary step is added with weighted GKE Gateway routing

    Frontend — Blue/Green

    • Frontend services run at low load, so the cost of maintaining a second idle slot is negligible. The benefit is instant rollback.
    • A Kubernetes Service selector change flips traffic from blue to green instantly (kubectl patch label selectors). This matters for frontend deployments where visual regressions are hard to catch in CI and need to be reverted the moment they're seen in production.

    Workers — Rolling with graceful drain

    Workers processing Pub/Sub messages or BullMQ jobs must never be killed mid-job. The implementation uses a

    • The preStop and terminationGracePeriodSeconds pauses new message consumption and waits for in-flight jobs to complete. SIGTERM is the signal to drain, not to abort.

    Rollback for On-Call / Non-Authors

    This is a 2 step process:

    1. Revert GitOps PR that bumped the image tag
    2. Open a new PR with the previous SHA

    This is a git operation, not a kubectl command, and it creates an audit trail.

    For immediate mitigation (faster than a PR review cycle):

    1. kubectl rollout undo deployment/<name> in the relevant namespace.
    2. GitOps PR within the same incident window to reconcile. This two-step approach balances speed with auditability.

    Decision: Runbook over tooling
    Like I mentioned before, I prefer keeping tooling lean. Fancy rollback UIs (ArgoCD rollback button, custom Slack bots) are tempting but add complexity and failure modes. A written runbook with two clear paths is more reliable and easier to execute under stress. On-call engineers should practice it in the non-prod environments quarterly.

    Deployment Traceability

    This links back to the image tagging I already mentioned. Every running pod must be traceable back to a commit and PR. This is achieved through a combination of:

    • Image tag = commit SHA, always. Any pod's image field is a direct link to the source commit.
    • Pod annotations set by the Helm chart: deploy.marketplace/commit-sha, deploy.marketplace/deployed-by, deploy.marketplace/gitops-pr.
    • FluxCD's last-applied-revision annotation on HelmRelease objects, which records the GitOps repo commit that applied the change.

    Secret Management

    Context & Problem

    What NOT to doStoring secrets in the GitOps repository, even encrypted with SOPS or Sealed Secrets. It creates an asymmetric security model where the GitOps repo becomes a target, and rotation requires a PR.

    Where Secrets Live

    All secrets are stored in GCP Secret Manager.

    Each secret is named by service: order-service/db-password. No need for environment-specific naming due to one GCP project per environment. This gives fine-grained IAM control.

    Each service's Workload Identity service account can only read its own secrets and provides automatic versioning, access logging, and rotation triggers via Cloud Pub/Sub notifications.

    Injection into Workloads

    Secret delivery flow. Secrets never touch the GitOps repo. ESO syncs from Secret Manager to in-cluster Kubernetes Secrets, which pods consume as env vars.

    External Secrets Operator (ESO) runs in the cluster and watches ExternalSecret custom resources. Each service declares what secrets it needs. ESO fetches them from Secret Manager and creates or updates the corresponding Kubernetes Secret. The sync interval is configurable — 1 minute for most secrets, shorter if the service supports hot reload.

    Pods then consume these as environment variables via envFrom. For secrets that need file-based delivery (TLS certs, service account keys), ESO can also write them as mounted files. The Helm chart template handles this so service teams never touch the ESO resource directly. Teams declare keys in values.yaml and the chart generates the ExternalSecret.

    Rotation Without Downtime

    Secret rotation without downtime requires the application to support hot-reload or the infrastructure to perform a rolling restart. The approach depends on the secret type:

    • Database credentials: MongoDB Atlas supports dual-active credentials during rotation. Add the new credential, update Secret Manager, ESO syncs the Kubernetes Secret within the refresh interval, a rolling restart picks up the new value, then revoke the old credential. Zero Downtime
    • API keys / JWT secrets: Services should read these from env vars, not cache at startup. If they do cache, a rolling restart is triggered automatically by an annotation change on the Deployment (the Helm chart can include a hash of the secret as an annotation, which changes when the secret rotates, triggering a rollout).
    • Automatic rotation triggers: Secret Manager supports Pub/Sub notifications on secret versions. A Cloud Function (or Cloud Run job) subscribes to rotation events, updates the Kubernetes Secret directly via the ESO sync, and optionally triggers a rolling restart via the Kubernetes API if the service can't hot-reload.
    What I'd skip
    I currently use HashiCorp Vault and it's a common recommendation, but adds a significant operational component. GCP Secret Manager provides equivalent functionality for a GCP-native stack with zero operational overhead. Vault makes sense if you have multi-cloud requirements or need its dynamic secret generation capabilities (short-lived database credentials). Neither applies here. SOPS + Age for encrypting secrets in the GitOps repo is also skipped.

    Developer Experience & Platform Self-service

    Context & Problem

    Platform teams that own too much become a bottleneck for the engineers they're supposed to support. If engineers need to open a ticket to change a replica count, add an environment variable, or investigate a production log, the platform team spends its time on meanial tasks instead of improving the product. The goal is to define a clear boundary: what service teams own, what the platform team owns, and what neither team should need to touch manually.

    What Service Teams Own — No Platform Involvement Needed

    CapabilityMechanismNotes
    Scaling configuration Edit values.yaml in GitOps repo, open PR HPA min/max and target CPU are in the values file. Teams own these.*
    Environment variables (non-secret) Edit env block in values.yaml Goes through PR for traceability. No platform team involvement.
    Secret references Add key to envFromSecrets in values.yaml Platform team pre-provisions the GSA permissions; team adds the reference.
    Log queries GCP Cloud Logging / Grafana Loki directly Service teams have read access to their namespace's logs. No tickets needed.
    Triggering a rollback Revert GitOps PR or kubectl rollout undo via runbook All on-call engineers have the necessary RBAC permissions.
    Viewing metrics / dashboards Grafana (read access for all engineers) Service-level dashboards are pre-provisioned by the platform team for the common cases.
    Feature Flags?
    I noticed Feature Flags were mentioned. Feature flags are part of the application logic, not the platform configuration. If the platform team owns the feature flag system, then service teams need to involve them for any new flag, which creates friction and delays. Instead, I prefer to provide a recommended open-source feature flag library (like Unleash or Flagsmith) that teams can integrate into their codebase independently.

    What the Platform Team Owns

    • Cluster provisioning and upgrades (Terraform + GKE channel management)
    • The shared Helm chart and its defaults
    • Namespace creation and IAM/RBAC setup for new services
    • Network policies, admission policies and *Resource quotas
    • The observability stack (Prometheus, Grafana, alerting rules)
    • Secret Manager structure and GSA permission templates
    • KEDA and ESO operator upgrades
    • The CI reusable workflow
    Boundary principle
    If a service team needs the platform team to change a replica count or add an env var, that's a platform failure, not a normal workflow. The test: can a new service be deployed entirely without a Slack message to the platform team? If not, fix the self-service gap.

    The One Thing to Build First

    The highest-leverage single investment is a service scaffolding CLI integrated into the Nx monorepo. When a team creates a new service, they run one command and get a fully-wired project: Dockerfile, Helm values file with sensible defaults, CI caller workflow, ExternalSecret template, and a basic /health endpoint. The generated service is ready to deploy to dev without the team ever reading a Kubernetes manifest.

    # What a team runs to add a new service
npx nx generate @marketplace/service-generator:service \
  --name=inventory-service \
  --type=api \
  --port=3001

    This generator should output:

    • apps/inventory-service/ — NestJS scaffold with health endpoint
    • apps/inventory-service/Dockerfile
    • .github/workflows/inventory-service.yml — calls the reusable CI workflow
    • gitops/envs/dev/apps/inventory-service/values.yaml — pre-filled with name, port, and type
    • A PR description template explaining what to change before going to prod

    The reason this comes first: it multiplies every other platform investment. Good defaults in the generator mean every new service inherits the resource configuration, security context, and health check setup automatically. Bad defaults proliferate just as quickly, so getting this right early pays compound dividends.

    Observability & Incident Readiness

    Context & Problem

    Prometheus and Grafana are already in the stack, which is a good starting point. The problem at scale is alert fatigue. As services multiply, so do dashboards and alerts, until on-call is a wall of noise that nobody takes seriously. The design goal is to keep alerts actionable regardless of service count, and to make debugging fast enough that a small on-call team doesn't burn out.

    What to Monitor and Alert On

    high priority = Immediate action required (eg. PagerDuty)

    medium priority and low priority = Investigation needed, but not urgent (eg. Slack)
    SignalRouting
    Error rate (5xx / total requests) high priority
    Pub/Sub subscription backlog high priority
    Pod crash loops high priority
    HPA at max replicas medium priority
    Certificate expiry low priority
    Failed deployments (FluxCD) medium priority
    MongoDB Atlas connection pool saturation medium priority

    The critical discipline is keeping the pager list short.

    As service count grows, alerts are grouped by service SLO tier, not by service name. A tier-1 service (payment processing, order placement) has tighter thresholds and wakes people up. A tier-3 service (recommendation engine) might degrade gracefully and alert to Slack. This keeps the pager manageable without ignoring real problems.

    Debugging a Latency Spike — End to End

    Latency spike debugging workflow. Each step uses a specific tool; the trace ID is the thread connecting them.

    Every log line includes the trace ID from the incoming request. When Grafana shows a latency spike on a service, the engineer opens Cloud Trace, filters by high-latency traces on the service, picks one, and follows the span waterfall to find the slow dependency. The log correlation by trace ID then shows the actual error message or slow query.

    The Grafana deployment marker overlay shows a vertical line on every metric chart when a deployment event occurs.

    Sustainable On-Call for a Small Team

    • Alert ownership: every alert is owned by a specific team, not "the platform team." If an alert fires with no clear owner, it gets reviewed in the weekly ops meeting and either assigned or deleted.
    • Runbooks next to alerts: every PagerDuty alert has a runbook link in its body. A short document with: what this alert means, the first three things to check, how to mitigate, and who to escalate to. Engineers update runbooks after incidents.
    • Post-mortems, not blame: blameless post-mortems with action items filed as tickets. The goal is making the next on-call shift easier, not reviewing what went wrong.
    • Noise reviews: monthly review of alert volume. Any alert that fires more than once without resulting in an action gets its threshold adjusted or removed.
    • Alert inhibition: if the cluster is in a maintenance window or a known degraded state, alerts are suppressed at the Alertmanager level rather than waking people up for known issues.

    On-Demand / Ephemeral Environments

    Current Approach Issues
    The ephemeral environments I currently work are a big topic of contention, but also a big learning opportunity. Our microservices have become too tightly coupled, so any change requires deploying the entire stack to individual dev environments. This leads to long wait times, noisy testing, and high costs.

    Proposed Design

    Ephemeral environment lifecycle. Environments are namespaces in the int cluster, not full clusters. TTL ensures teardown even if the PR is forgotten.

    Ephemeral Lifecycle

    1. Github Action triggers on PR labeled with preview
    2. Creates namespaces per PR within the existing int cluster.
    3. Deploy only changed services based on Nx's affected command, not the full 15-service stack.
    4. Closed PR = delete namespace

    Cost management

    Unmodified services are shared from the permanently running dev namespace via a service proxy or Kubernetes ExternalName service pointing at the dev deployments. This keeps the per-PR cost to roughly 30–50% of a full environment rather than 100%.

    A TTL of 72 hours is enforced by a CronJob that scans for namespaces with a preview.marketplace/expires-at annotation older than now and deletes them. This catches forgotten environments from stale PRs.

    Data seeding

    Ephemeral environments connect to a shared, read-only anonymized snapshot of the dev MongoDB Atlas cluster, not to a per-environment database. Creating a fresh database per environment is slow (2–5 minutes for data seeding) and expensive. The snapshot is refreshed weekly and contains realistic but fake data generated by a seeding script. Services that need to write during testing get a transient in-memory MongoDB instance (via the mongodb operator or a StatefulSet) seeded from the snapshot on startup.

    Trade-off: Shared infra vs full isolation

    Full isolation One Redis, One MongoDB, One Pub/Sub project per environment gives the cleanest test surface but adds 5–8 minutes to environment creation and significant cost.

    SharedFor most feature branches, sharing stateless infra (Redis, Pub/Sub topics in dev, read-only MongoDB) is fine.

    MitigationThe risk is test pollution from concurrent PR environments writing to the same dev topics. Use a PR-specific Pub/Sub subscription prefix (pr-123-) to avoid reading from the same subscription as other environments.
